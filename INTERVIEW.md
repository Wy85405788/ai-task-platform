### 1. 架构级：预分配 ID 模式 (Pre-allocation Pattern)
这是你今天解决最核心问题的方案。
* **工作中**：在处理长耗时任务（如 AI 生成、视频转码、支付回调）时，绝对不能等任务结束再存库。
* **面试话术**：*“在处理 SSE 流式输出时，我采用了‘先占位、后填充’的策略。在流开启前先通过一个原子请求在数据库创建记录并返回 ID。这样即使生成过程中途断开，系统也能通过 ID 追踪到这条残缺的任务，保证了状态的可追溯性，避免了丢单或数据孤岛。”*


### 2. 后端级：异步环境下的数据一致性 (Async DB Integrity)
你今天亲手操刀了 `SQLAlchemy` 的异步操作。
* **核心知识点**：`commit()` 与 `refresh()` 的配合。
* **面试考点**：*“在使用异步 ORM（如 SQLAlchemy 2.0）时，为什么 commit 之后还要 refresh？”*
* **深度回答**：因为 `commit` 只是把数据推到了数据库。在异步上下文里，Python 对象并不会自动同步数据库生成的自增 ID 或默认值。通过 `await db.refresh(obj)`，我们可以强制触发一次查询，把真实的数据库 ID 取回内存，确保后续的业务逻辑（比如给前端回传 ID）拿到的是真数据。

### 3. 前端级：类型驱动与防御性编程 (Type-Driven Development)
你给 Store 穿上了 TypeScript 的“防弹衣”。
* **核心知识点**：`Interface` 对齐与非空断言 (`!`)。
* **工作价值**：通过定义 `ITask` 接口，你确保了前端对 `description`、`status` 等字段的调用是安全的。
* **面试话术**：*“我通过 TypeScript 的接口（Interface）与后端 Pydantic 模型实现了强一致性契约。在处理异步获取的对象时，我利用‘类型收窄’（Type Narrowing）技术（如 if 判断或非空断言），在编译阶段就排除了 Cannot read property of null 这种常见的运行时崩溃隐患。”*

### 4. 数据库级：平滑迁移与数据清洗 (Data Migration)
你手动“洗数据”的那个过程，其实是数据库工程师的日常。
* **核心知识点**：临时表迁移法。
* **职场经验**：永远不要在生产环境直接修改主键 ID。正确的姿势是：**备份 -> 创建结构相同的临时表 -> 导入数据（不带旧 ID） -> 重命名交换**。这种“影子表”切换的思维，是实现“零停机迁移”的基础。
